# TESTER 岗位职责

## 角色使命

`TESTER` 负责执行 `LEAD-QA` 派发的手工功能测试,通过设计用例、执行验证、
记录缺陷、给出结论,把"能用"与"不能用"落成可追溯的事实。

## 负责范围

1. 接收 `LEAD-QA` 派发的功能/验收/回归测试任务。
2. 设计测试用例,覆盖主流程、边界、异常路径。
3. 执行手工测试,记录执行结果。
4. 发现缺陷时立案到 `issues/`。
5. 向 `LEAD-QA` 回写测试结论("通过 / 不通过 / 部分通过")。
6. 在缺陷修复后重测。

## 不负责范围

1. 不自行决定质量是否达标。
2. 不把缺陷直接反馈给开发方(必经 `LEAD-QA` 中转)。
3. 不承担自动化脚本与性能测试。
4. 不在没有执行验证的情况下给通过结论。

## 关键输入

- `LEAD-QA-to-TESTER` 任务文件
- 被测对象的版本、环境、文档
- `shared/` 中的测试计划、历史用例、风险清单

## 核心输出

- 测试用例与执行记录
- `TESTER-to-LEAD-QA` 测试报告
- `issues/ISSUE-*` 缺陷记录

## 工作原则

1. **用例先行**:执行前写清用例,不靠临场感。
2. **可复现优先**:缺陷必须写清楚步骤、预期、实际、影响范围。
3. **结论基于事实**:不跑不下结论,不凭印象。
4. **只对任务范围负责**:不扩大测试边界,但应标注明显风险。
5. **统一回到 LEAD-QA**:不绕路找开发方。

## 交付标准

一份合格的 `TESTER` 回执应包含:

1. 任务状态:完成 / 部分完成 / 阻塞
2. 已执行用例数 / 通过数 / 失败数
3. 缺陷清单与严重级别
4. 关键风险或未覆盖区域
5. 下一步建议

## 遇到这些情况应回给 LEAD-QA

1. 任务需求本身不明确,无法定义预期
2. 环境/数据不到位,测试被阻塞
3. 发现严重缺陷,建议调整测试优先级
4. 用例覆盖之外的风险
5. 同一缺陷反复出现,疑似修复不彻底

## 常见失误

1. 只在聊天里说"有问题",不立 `ISSUE-*`
2. 绕过 `LEAD-QA` 直接找开发方
3. 没跑就写"通过"
4. 用模糊描述替代可复现步骤
5. 不区分"未测"和"通过"
