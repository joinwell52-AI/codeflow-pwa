## 0. 根公理（Root Axiom）

拟写进 `codeflow-core.mdc` 顶部，作为所有规则的**总纲**：

> **Files are the protocol. Folders are the organization.**
> **文件即协议，文件夹即组织。**

一切具体条款都是这条公理在不同层级（角色 / 团队 / 项目 / 联邦）下的落地。

## 2. 暴露的协议漏洞（原 v1 四条 + 扩展两条）

### 漏洞 1：没有 thread_key 认领锁
任何 Agent 读到 pending 任务都能直接执行。两个 PM 会话同时在同一 thread_key 上作业时，无机制声明"当前线程已被 Session X 占用"。

**风险**：多 PM 并发编辑同一份 REPORT/TASK 文件会静默覆盖；派发重复下游任务。

### 漏洞 2：没有 ADMIN 显式授权门
PM 的"承诺 T+24h 交付"被下一个 PM 会话读成了"GO 信号"。协议里没有任何文件声明"ADMIN 已明确授权 PM 开始执行"。

**风险**：任何承诺都会被误读为授权，ADMIN 失去对执行节奏的控制。

### 漏洞 3：新会话默认行为是"冲"而非"看"
当前 Core Principle + 欢迎语鼓励 Agent 上线就做事。缺少"首要动作是观察与汇报，不是执行"的硬约束。

**风险**：ADMIN 每开一个新会话都可能触发意外的执行动作。

### 漏洞 4：文件里无会话身份标识
frontmatter 只写 `sender: PM`，不写是哪个会话的 PM。两个 PM 会话的产出无法区分，审计不可行。

**风险**：事后无法追溯"谁写的这份 REPORT"、"为什么同一 thread 有两份 ADMIN-to-PM"。

### 漏洞 5：项目身份配置不是唯一权威源（Bridgeflow 现场暴露）
`codeflow.json` / `.cursor/rules/*.mdc` / `docs/agents/tasks/` 三者对"本项目团队是什么、角色有哪些"各说各话时，Agent 无从判断，只能凭读取顺序瞎蒙。

**风险**：身份歧义扩散到所有下游行为；Solo/Team 模式无法可靠判定。

### 漏洞 6：缺少跨项目/跨团队的协议边界定义
FCoP 目前假设所有 Agent 共享同一个 `docs/agents/` 文件夹。真实场景里：
- 单项目有多团队协作（dev-team + data-team）
- 多项目有对接工程（project-a 的 PM 要派工给 project-b 的 DISPATCHER）

**风险**：跨边界时 Agent 不知道该把文件写哪里、该从哪里读 inbox。

## 3. 建议的协议补丁（8 条款）

### 条款 0 · 角色前置 / UNBOUND 协议（**核心**）

Agent 启动时默认处于 **UNBOUND 态**，必须按以下顺序执行，不得跳过、不得交叉：

#### 步骤 1：只读扫描（不改、不删、不创建、不写）

按固定顺序读取**客观事实**：

| 来源 | 读什么 | 用途 |
|---|---|---|
| `docs/agents/codeflow.json`（或 `{team}/team.json`） | `mode` / `team` / `roles` / `leader` | 判定 solo 还是 team、有哪些角色 |
| `.cursor/rules/*.mdc`（或等效系统提示） | 文件名 + frontmatter | 知道项目定义了哪些角色规则 |
| `docs/agents/tasks/` | **仅文件名，不读正文** | 知道哪些 thread_key 活跃、谁在等谁 |
| `docs/agents/reports/` | 仅文件名 | 知道哪些线程已结案 |
| `codeflow-core.mdc` 或等效系统提示 | 全文 | 知道协议本身 |

**硬约束**：
- ❌ 不得读取任何任务正文（防止被任务内容污染为某角色）
- ❌ 不得修改任何文件
- ❌ 不得调用写类工具（`write_task` / `drop_suggestion` / `archive_task` 等）
- ✅ 只允许 `list_*` / `read_*` 类只读工具，且 `read_*` 仅用于读配置/规则文件

#### 步骤 2：客观汇报（纯事实、零解读）

汇报格式**强制固定**，Agent 不得发挥：

```markdown
## UNBOUND 汇报 · 项目：{path}

- codeflow.json 存在：是/否
  - mode: {solo | team | 未声明}
  - team: {team_name | 未声明}
  - 已声明角色: [PM, DEV, QA, OPS] 或 [未声明]
- .cursor/rules/ 下存在的角色规则文件: [pm-bridge.mdc, dev-bridge.mdc, ...]
- docs/agents/tasks/ 下活跃线程（按 thread_key 去重）:
  - dev-team/anti_hang_triage_20260421 · 最后一份: TASK-001-PM-to-ADMIN.md · 状态: 等 ADMIN
  - ...
- 协议版本: FCoP v1.1 (codeflow-core.mdc hash: abc123)

## 我的身份状态：UNBOUND（未指派）

## 请 ADMIN 指派：
请用以下格式告诉我身份：
> 你是 {ROLE}，在 {team}，线程 {thread_key}（可选）
例如：
> 你是 PM，在 dev-team，线程 anti_hang_triage_20260421
```

#### 步骤 3：等待（Wait）

- 不轮询、不催促、不额外读文件
- 收到 ADMIN 符合格式的指派 → 转入 **BOUND 态**，从此以该身份行动
- 收到其他内容（闲聊、问题、任务描述）→ 礼貌复述："我仍处于 UNBOUND 态，请先指派身份。"

**BOUND 后**：`sender` 字段永久锁定为指派角色，切换角色必须重开会话。

### 条款 1 · Solo vs Team 模式显式声明

`codeflow.json` 新增必选字段 `mode`：

```json
{
  "mode": "solo" | "team",
  "team": "dev-team",       // mode=team 时必填
  "roles": [...]            // mode=team 时必填
}
```

| 模式 | 特征 | 文件要求 | "落成文件"原则 |
|---|---|---|---|
| **Solo** | ADMIN 对 1 个 Agent，无跨角色协作 | 可不走 `tasks/` 全流程，维护 `drawer/` + 必要 REPORT 即可 | **仍然生效** — 单角色自审（提案/审查分离）必须落文件 |
| **Team** | 多角色协作 | 完整 FCoP：TASK + REPORT + ISSUE | 跨角色交接**全部**走文件 |

Solo 模式下，Rule 0 的团队字段填 `"solo"`，Agent 不会冒失跳出来认领 PM-to-DEV 类文件。

---

### 条款 2 · 团队作用域目录（Level 2 组织）

单团队项目保持平铺（兼容现状），多团队项目升级为团队子目录：

```
# 单团队（当前 Bridgeflow 结构，不迁移）
docs/agents/
├── codeflow.json
├── tasks/
├── reports/
├── issues/
└── log/

# 多团队
docs/agents/
├── codeflow.json            ← 声明本项目有哪些 team
├── dev-team/
│   ├── team.json            ← 本 team 的 mode / roles / leader
│   ├── tasks/ reports/ issues/ log/
└── media-team/
    ├── team.json
    ├── tasks/ reports/ issues/ log/
```

**识别规则**：
- `docs/agents/tasks/` 存在 → 单团队模式
- `docs/agents/{team_name}/tasks/` 存在 → 多团队模式

**命名约束**：`thread_key` 必须以团队名为前缀，防止跨 team 误认领：

```yaml
thread_key: dev-team/anti_hang_triage_20260421
```

**角色绑定**：一个 Agent 会话 **只能** 绑定一个 `{team}/{role}`；要换团队/角色必须重开会话。

### 条款 3 · codeflow.json 是项目身份的唯一权威源

当 `codeflow.json` / `.cursor/rules/` / `docs/agents/tasks/` 三者冲突时：

1. **`codeflow.json` 永远胜**
2. 其他两者被视为需要修正的不一致项
3. UNBOUND 汇报必须列出检测到的不一致，由 ADMIN 裁决

本条在 Bridgeflow 上的落地证据：本次已把 `codeflow.json` 从 `mvp-team` 改为 `dev-team`，对齐实际运行的规则与任务。

---

### 条款 4 · thread_key 单活跃 PM 原则

同一 `thread_key` 在任一时刻**只能**有一个活跃的 PM 会话在执行。

**判定**：若该 thread_key 下存在未结案的 `*-PM-to-ADMIN.md` 接单回执，则视为"已有活跃 PM"。

**后加入的 PM 会话行为**：
- 默认转为**协审者（Observer/Co-reviewer）**，不抢主导权
- 可以读、可以点评、可以提 `drop_suggestion`
- **不得**编辑同 thread_key 下已有任务的延续文件（即不得写 `TASK-00X-PM-to-DEV.md`）

**主导权转移**：只能由 ADMIN 显式声明：
> 当前会话接管 `thread_key` 的 PM 主导权，原会话转为协审。

---

### 条款 5 · frontmatter 加 session_id（可选但建议）

frontmatter 增加可选字段 `session_id`（由 Agent 生成的短字符串，如 `cs-240421-171851-a2b3`）：

```yaml
---
protocol: fcop
version: 1
sender: PM
session_id: cs-240421-171851-a2b3   # 可选
recipient: DEV
thread_key: dev-team/anti_hang_triage_20260421
priority: P2
---
```

**用途**：
- 事后审计同一角色多会话的产出来源
- 同一 thread 下多会话并存时，从文件层面区分"哪个 PM 写的"
- 对协议行为无硬约束，纯追溯用

---

### 条款 6 · 跨项目/跨团队协议边界（inbox / outbox）

项目内的 `docs/agents/` 增加两个系统目录：

```
docs/agents/
├── {team}/                  ← 内部协作
├── outbox/                  ← 对外发件
│   ├── to-{other-project}/
│   └── to-{other-team}/
└── inbox/                   ← 对外收件
    ├── from-{other-project}/
    └── from-{other-team}/
```

**协议保证**：处理 `inbox/from-*/TASK-*.md` 的规则与处理 `{team}/tasks/TASK-*.md` **完全同构**。Agent 不需要因为来源不同而改变行为。

**同步机制与协议解耦**：outbox → 对方 inbox 的搬运可以用：
- Git 仓库联动（commit + pull）
- 中继服务器（`server/relay/` 扩展）
- 文件同步工具（rsync / 网盘）

**FCoP 不指定具体同步方式**，只要求"文件到达对方 inbox 之后行为一致"。

---

### 条款 7 · URI 路由标注（可选，与条款 6 配合）

在 frontmatter 里用 `fcop://` URI 表达完整路径，用于跨作用域寻址：

```yaml
sender:    fcop://project-a/dev-team/PM
recipient: fcop://project-b/data-team/DISPATCHER
thread_key: cross/a-to-b/integration_20260421
```

URI **不取代文件位置**，只是给 MCP/工具一种**与物理位置解耦**的查询语法。物理文件仍在 outbox / inbox。
