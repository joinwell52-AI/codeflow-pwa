# BridgeFlow Configuration Reference

**Version:** v0.2.1

---

## Configuration Files Overview

| File | Location | Purpose |
|------|----------|---------|
| `bridgeflow_config.json` | PC working directory (generated by `bridgeflow init`) | All PC execution machine runtime parameters |
| `web/pwa/config.js` | PWA static resources | Phone-side relay address, cache version |

---

## 1. `bridgeflow_config.json` Full Parameter Dictionary

### Generation & Update

```powershell
bridgeflow init                          # First-time generation (defaults)
bridgeflow init --relay-url wss://...    # Specify relay URL during generation
bridgeflow init --room-key my-room       # Specify room key during generation
bridgeflow init --force                  # Force overwrite (⚠️ resets binding state)
```

When a config already exists, `bridgeflow init --relay-url/--room-key` only updates the corresponding field without affecting others.

---

### `relay` — Relay Connection

| Field | Type | Default | Required | Description |
|-------|------|---------|----------|-------------|
| `url` | `string` | `"wss://ai.chedian.cc/bridgeflow/ws/"` | ✅ | WebSocket relay URL. Must be `wss://` or `ws://`. Change when using self-hosted relay |
| `room_key` | `string` | `"bridgeflow-default"` | ✅ | Room isolation key. PC and phone must use the **exact same value** to communicate. Each team should use a unique random string |
| `shared_secret` | `string` | `""` | ❌ | Shared secret (reserved for future use, currently inactive) |

> **`room_key` Security Note:** The default `bridgeflow-default` is a public room — anyone can join. For team use, change it to a random string, e.g. `team-abc-2026-x7k`.

```json
"relay": {
  "url": "wss://ai.chedian.cc/bridgeflow/ws/",
  "room_key": "your-private-room",
  "shared_secret": ""
}
```

---

### `device` — Device Identity

| Field | Type | Default | Required | Description |
|-------|------|---------|----------|-------------|
| `device_id` | `string` | hostname slug | ✅ | Unique PC identifier for relay routing. Auto-generated from hostname by `bridgeflow init`, e.g. `my-pc`. Each PC must be different when using multiple machines |
| `device_name` | `string` | `"{hostname} AI Executor"` | ✅ | Display name shown on dashboard and phone binding info. Can be manually changed |
| `owner_role` | `string` | `"PM01"` | ✅ | Primary AI role assigned to this machine. Options: `PM01` / `DEV01` / `OPS01` / `QA01` |
| `machine_code` | `string` | `"BF-{12-char hash}"` | ✅ | Machine code for QR scan binding, format `BF-XXXXXXXX`. Generated from hostname + MAC address hash. **Do not manually modify** |

```json
"device": {
  "device_id": "my-pc",
  "device_name": "My AI Executor",
  "owner_role": "PM01",
  "machine_code": "BF-A3F2C8D1E2F3"
}
```

### `bind` — Device Binding State

> ⚠️ This section is auto-managed by CLI commands. **Manual modification is not recommended.**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `status` | `string` | `"unbound"` | Binding state. Options: `unbound` / `pending` / `bound` |
| `bound_mobile_device_id` | `string` | `""` | Bound phone device ID (auto-filled after binding) |
| `bound_mobile_device_name` | `string` | `""` | Bound phone device name |
| `bound_at` | `string` | `""` | Binding timestamp, format `YYYY-MM-DD HH:mm:ss` |
| `bind_code_ttl_seconds` | `integer` | `600` | One-time bind code validity (seconds). Default 600s (10 minutes) |
| `pending_bind_code` | `string` | `""` | Pending one-time bind code (6-char alphanumeric) |
| `pending_bind_expires_at` | `string` | `""` | Bind code expiration time |
| `pending_mobile_device_id` | `string` | `""` | Pending phone device ID |
| `pending_mobile_device_name` | `string` | `""` | Pending phone device name |
| `last_bind_code_issued_at` | `string` | `""` | Last bind code issuance time |

```json
"bind": {
  "status": "unbound",
  "bound_mobile_device_id": "",
  "bound_mobile_device_name": "",
  "bound_at": "",
  "bind_code_ttl_seconds": 600,
  "pending_bind_code": "",
  "pending_bind_expires_at": "",
  "pending_mobile_device_id": "",
  "pending_mobile_device_name": "",
  "last_bind_code_issued_at": ""
}
```

### `project` — Project Directory

| Field | Type | Default | Required | Description |
|-------|------|---------|----------|-------------|
| `root_dir` | `string` | `.` (absolute path of working directory) | ✅ | Project root. Auto-set by `bridgeflow init` to the directory where the command is run |
| `tasks_dir` | `string` | `"docs/agents/tasks"` | ✅ | Task file directory, relative to `root_dir`. Tasks are written here when AI receives assignments |
| `reports_dir` | `string` | `"docs/agents/reports"` | ✅ | Reply/report directory, relative to `root_dir`. AI writes reports here upon completion |
| `issues_dir` | `string` | `"docs/agents/issues"` | ✅ | Issue record directory, relative to `root_dir` |

```json
"project": {
  "root_dir": "C:\\Users\\YourName\\my-ai-team",
  "tasks_dir": "docs/agents/tasks",
  "reports_dir": "docs/agents/reports",
  "issues_dir": "docs/agents/issues"
}
```

> **Note:** `root_dir` is auto-filled by `bridgeflow init` based on your current directory. No manual setup needed — works on any drive letter, any path.

### `runtime` — Runtime Parameters

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `runtime_dir` | `string` | `".bridgeflow/runtime"` | Runtime state file root (excluded from `.gitignore`) |
| `status_dir` | `string` | `".bridgeflow/runtime/status"` | Role status file directory (JSON files recording online/busy/idle state) |
| `task_details_dir` | `string` | `".bridgeflow/runtime/task_details"` | Task details cache directory (for dashboard and phone queries) |
| `idle_seconds` | `integer` | `180` | Seconds of inactivity before a role is marked "idle" |
| `stale_task_seconds` | `integer` | `900` | Seconds without reply before a task is marked "stale/blocked" |

```json
"runtime": {
  "runtime_dir": ".bridgeflow/runtime",
  "status_dir": ".bridgeflow/runtime/status",
  "task_details_dir": ".bridgeflow/runtime/task_details",
  "idle_seconds": 180,
  "stale_task_seconds": 900
}
```

### `patrol` — Patrol Parameters

Patrol is the mechanism where the PC periodically checks each AI role's task status and sends reminders.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `message` | `string` | `"巡检"` | Patrol trigger text sent to Cursor role windows |
| `poll_interval` | `integer` | `10` | Patrol polling interval (seconds) |
| `check_delay` | `integer` | `15` | Seconds to wait after sending patrol message before checking response |
| `max_retry` | `integer` | `3` | Max retries per patrol round |
| `confidence` | `float` | `0.7` | Image matching confidence threshold (0.0–1.0) for window recognition |
| `templates_dir` | `string` | `"ops/patrol_templates"` | Patrol image template directory |
| `all_worker_chats` | `array` | `["1-PM", "2-DEV", "3-QA", "4-OPS"]` | All role window names participating in patrol (**fixed, do not change**) |
| `role_to_chat` | `object` | `{"PM01":"1-PM","DEV01":"2-DEV","QA01":"3-QA","OPS01":"4-OPS"}` | Role ID to Cursor Agent name mapping (**fixed, do not change**) |

```json
"patrol": {
  "message": "巡检",
  "poll_interval": 10,
  "check_delay": 15,
  "max_retry": 3,
  "confidence": 0.7,
  "templates_dir": "ops/patrol_templates",
  "all_worker_chats": ["1-PM", "2-DEV", "3-QA", "4-OPS"],
  "role_to_chat": {
    "PM01": "1-PM",
    "DEV01": "2-DEV",
    "QA01": "3-QA",
    "OPS01": "4-OPS"
  }
}
```

## 2. `web/pwa/config.js` Parameter Dictionary

The phone PWA reads connection parameters from this file. **Changes must be pushed to GitHub Pages to take effect.**

| Field | Type | Default | Required | Description |
|-------|------|---------|----------|-------------|
| `appName` | `string` | `"BridgeFlow"` | ✅ | Application name, shown in PWA title and home screen icon |
| `appVersion` | `string` | `"1.6.0"` | ✅ | **Service Worker cache version.** Must be incremented whenever any PWA file changes, otherwise phones see the old version |
| `relayUrl` | `string` | `"wss://ai.chedian.cc/bridgeflow/ws/"` | ✅ | Relay WebSocket URL, must match PC-side `relay.url` |
| `relayLabel` | `string` | `"Public Relay"` | ❌ | Relay display name, shown in the "My" page configuration info |
| `roomKey` | `string` | `"bridgeflow-default"` | ✅ | Room key, must exactly match PC-side `relay.room_key` |
| `autoConnect` | `boolean` | `true` | ❌ | Auto-connect to relay when opening PWA. `true` recommended |
| `defaultTarget` | `string` | `"PM"` | ❌ | Default role displayed in task list. Options: `"PM"` / `"DEV"` / `"OPS"` / `"QA"` |

```js
global.BRIDGEFLOW_CONFIG = {
  appName:       "BridgeFlow",
  appVersion:    "1.6.0",
  relayUrl:      "wss://ai.chedian.cc/bridgeflow/ws/",
  relayLabel:    "Public Relay",
  roomKey:       "bridgeflow-default",
  autoConnect:   true,
  defaultTarget: "PM"
};
```

> **Key Tip:** `appVersion` format should ideally stay in sync with the Python package version, but they are independent — PWA-specific changes also require separate version bumps.

## 3. `bridgeflow init` Command Parameters

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `--project-root PATH` | `string` | Current directory | Root directory for config and task files |
| `--relay-url URL` | `string` | `wss://ai.chedian.cc/bridgeflow/ws/` | Override relay URL |
| `--room-key KEY` | `string` | `bridgeflow-default` | Override room key |
| `--force` | `flag` | `false` | Force regenerate config (⚠️ resets binding state and machine code) |

---

## 4. `bridgeflow run` Command Parameters

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `--config PATH` | `string` | `./bridgeflow_config.json` | Specify config file path |

## 5. Task Priority Enum

| Value | Meaning | Expected Response |
|-------|---------|-------------------|
| `P0` | Critical | Handle immediately, reply within 1 hour |
| `P1` | High | Complete within the day |
| `P2` | Medium | Complete within the week |
| `P3` | Low | Handle when available |

---

## 6. Role ID Enum

| Role ID | Type | Description |
|---------|------|-------------|
| `ADMIN01` | Human | Phone-side operator (you) |
| `PM01` | AI | Project Manager — task decomposition and dispatch |
| `DEV01` | AI | Developer — code implementation |
| `OPS01` | AI | Operations Engineer — deployment and server operations |
| `QA01` | AI | QA Engineer — testing and quality verification |
