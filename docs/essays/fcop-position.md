# FCoP 的定位：一种文件系统层的协作协议

> **What FCoP is**: not another agent framework; a coordination protocol
> that lives on the filesystem surface, from which governance, audit,
> and multi-agent collaboration emerge as upper-layer consequences.

**作者**：CodeFlow 团队 · 2026-04-22
**关键词**：FCoP, protocol-first, filesystem-native, filename-as-routing-key, protocol standards, multi-agent governance

---

## 1. 一句话定位

> **FCoP 是一种以文件系统为载体的协作协议；治理、审计、多 Agent 调度，
> 是它自然长出的上层形态，而不是它本身。**

这个定位比早期一些表述（"AI 团队治理协议"、"多 Agent 组织语法"）
更**底层**，也更**稳**。原因在后面几节展开。

---

## 2. 坐标：协议先 vs 运行时先

多 Agent 系统有两种物种，取决于**协议和运行时谁先存在**：

| 运行时先（Runtime-first） | 协议先（Protocol-first） |
|---|---|
| Servlet API / J2EE 容器 | HTTP |
| Perforce | Git |
| 专有邮件系统 | SMTP + Maildir |
| 专有新闻 API | RSS |
| LangGraph / CrewAI / AutoGen / Temporal | **FCoP** |

**左列**：先有运行时，再在运行时里定义一套"怎么交互"的约定。离开
运行时，约定就失效。

**右列**：协议在任何实现之前就已经完整可观察。**谁懂协议谁就能接入**，
不需要运行时。

历史反复证明：右列早期弱于左列（没生态、没工具、手动操作多），
**长期强于左列**（锁定成本近零、标准化后赢家通吃、跨厂商跨语言跨年代）。

FCoP 位于右列。这不是选择题；是**文件即协议**这个架构决定让它只能
在右列。

---

## 3. 机制：为什么"文件即协议"是底层真相

FCoP 的协议表面分布在四个地方，加起来 **90% 不在文件正文里**：

| 协议表面 | 比例 | 说明 |
|---|---|---|
| **文件名**（`TASK-YYYYMMDD-NNN-SENDER-to-RECIPIENT.md`） | ~40% | 路由键：发给谁、谁发的、第几号、哪一天 |
| **YAML frontmatter**（`thread_key` / `priority` / `type`） | ~30% | 线程归属、优先级、任务类型 |
| **目录语义**（`tasks/` / `reports/` / `issues/` / `log/`） | ~20% | 生命周期阶段：进行中 / 已回 / 待处理 / 已归档 |
| **行为约定**（Rule 0 UNBOUND、Rule 0.b 自审、巡检、建议通道） | ~10% | 运行时协议，无法完全外化到文件 |

前三项合计 ~90%——**协议长在文件系统表面**，这是以下一切能力的前提：

- 不读正文，`ls docs/agents/tasks/` 就知道现场
- 不打开文件，用文件名就能判断是不是发给自己的
- 不靠数据库，目录结构就是状态投影
- 不依赖 Python 运行时，任何能 `cat / grep / mv` 的工具（Agent、人、bash、其他 MCP）都是合法参与者

最后 10% 的行为约定需要 Agent 配合触发（比如 `unbound_report`），
但**前 90% 即使 Agent 消失也依然是协议**——这是 FCoP 和 Agent 框架
的物种级差别。

---

## 4. 精确表述

早期一些口号（"文件名即协议"）在感染力上很好，但不完全精确。
精确的表达是：

> **协议长在文件系统表面，行为约定由协议触发。**

这句话同时说明两件事：

1. 协议的**主体**在文件系统上（文件名 + frontmatter + 目录 = 90%）
2. 协议的**触发**依赖 Agent 遵守行为约定（主动 UNBOUND 汇报、主动落文件、主动不伪造）

第 1 点让 FCoP 独立于任何运行时；第 2 点让 FCoP 仍然需要 Agent
是"愿意配合的 Agent"——这不是弱点，是协议的**行为契约**，和 HTTP
需要 server 正确返回状态码一样。

## 5. 从这个定位自然推出的三层未来

从"文件系统层协议"这个基座出发，向上长出三层形态：

### 第一层：人 + 1 AI 的留痕协议（**今天就有价值**）

Solo 模式：1 个 ADMIN + 1 个 Agent（`ME`）。
两个角色就够跑 FCoP。任务 / 回执 / 归档 / 建议四条通道全开。

**价值**：哪怕未来"多 Agent 时代"是伪命题，这一层也站得住。
只要"人 + AI 协作时想要留证据"这个需求存在，FCoP 就有价值。

这是 FCoP 最**朴素也最稳**的价值命题——**退路定位**，不依赖任何
未来假设。

### 第二层：1 人 + 多 AI 的团队协作（**当前主战场**）

Team 模式：ADMIN + 多个 AI 角色（`PM / DEV / QA / OPS` 或自定义）。
`.cursor/rules/` 分发规则、`shared/roles/` 分发职责、`tasks/` 做路由、
`reports/` 做回执闭环。

**价值**：把多 Agent 协作的"现场"从 Agent 的短期上下文窗口**拽到
文件系统上**，让人和其他 Agent 都能读、能审、能回放。

这是 FCoP 现在最主要的落地场景。预设 4 套团队（dev-team /
media-team / mvp-team / qa-team）+ 自定义团队覆盖绝大多数配置。

### 第三层：多人 + 多 AI + 多组织的协作接口标准（**长线远景**）

如果 FCoP 的命名规则 / frontmatter 字段 / 目录语义真的被标准化
到"跨公司跨工具可识别"的程度，它可能扮演类似 HTTP / Git / RSS 的
基础设施角色：

- 不同公司的 Agent 把工作交接给对方时，FCoP 文件能续上
- 审计方不需要登录任何平台，拿着 `tasks/` + `reports/` 就能回放
- 开源项目用 FCoP 累积可展示的"AI 协作开发史"，成为协议的活案例

这一层对应的不是"Git 协议"，是"HTTP 协议"——基础设施级别。

能不能走到这一步不由协议决定，由**分发**和**采用**决定。

## 6. 工具化的边界：便利层不是真相层

FCoP 有一个 MCP 工具箱（`fcop` Python 包，目前约 19 个工具：
`init_project` / `write_task` / `read_task` / `archive_task` / ...）。
这是一个真实的设计取舍：

| 工具多 | 工具少 |
|---|---|
| Agent 用起来省事 | 协议更纯粹，接入成本更低 |
| 协议藏在 Python 代码里 | 协议暴露在文件系统上 |
| 新 Agent 必须学工具 | 新 Agent 看一眼 `ls` 就知道 |

**FCoP 的取舍是"两者都要，但分层清楚"**：

> **工具是便利层，不是真相层。**
>
> 每个 MCP 工具都必须能**完整拆解成 N 个文件操作**，这个拆解要写在
> 工具的 docstring 里。这样：
>
> 1. 没装 `fcop` MCP 的团队，靠 `ls / cat / mv / git` 也能参与协作
> 2. 装了 `fcop` MCP 的 Agent，用工具调用省事
> 3. 工具改动只是便利层演进，**底层协议永不绑定工具**

举例：

| 工具 | 本质的文件操作 |
|---|---|
| `write_task(recipient, body)` | 在 `tasks/` 下写一个符合 `_TASK_FILENAME_RE` 的 .md 文件 |
| `archive_task(task_id)` | `mv tasks/TASK-XXX* log/` |
| `list_tasks(recipient)` | `ls tasks/ | grep -E '...-to-RECIPIENT'` |
| `read_task(task_id)` | `cat tasks/TASK-XXX.md` |

**一个工具如果没法还原成几条文件操作，就不该加进 `fcop` 包**——
否则就是在把协议藏进运行时，违背 FCoP 的根本定位。

## 7. 和 LangGraph / Temporal / MCP 的分层

FCoP 不抢这些系统的位置，也不被它们包含。**不同栈位**：

```
┌─────────────────────────────────────────────────────────┐
│  应用层       用户的实际需求（做网页、做视频、做报表...）│
├─────────────────────────────────────────────────────────┤
│  协作层       FCoP —— 协议、角色、回执、责任链          │
│              （文件系统即载体）                          │
├─────────────────────────────────────────────────────────┤
│  编排层       LangGraph / Temporal                       │
│              （运行时状态机、重试、并发）                │
├─────────────────────────────────────────────────────────┤
│  互联层       A2A（Agent-to-Agent 网络协议，规划中）     │
├─────────────────────────────────────────────────────────┤
│  工具层       MCP（工具调用 / 资源访问的标准）           │
├─────────────────────────────────────────────────────────┤
│  模型层       Claude / GPT / Gemini / Llama / ...        │
└─────────────────────────────────────────────────────────┘
```

FCoP 在**协作层**。它不关心：

- 模型是什么（模型层的事）
- 工具怎么调（MCP 层的事）
- Agent 之间怎么通信（A2A 层的事）
- 任务流程怎么执行、怎么重试、怎么并发（LangGraph/Temporal 层的事）

它只关心：

- 这次协作"算不算真的发生过"
- 现场能不能被任何人（人或 Agent）读懂
- 历史能不能回放
- 责任能不能追溯

这四件事都要求协议**独立于任何运行时**——所以它必须长在文件系统层。

## 8. 边界与代价：文件系统协议也有物理极限

"文件系统即消息总线"不是免费的。历史上有几个同架构参照物踩过的墙：

| 先行者 | 撞过的墙 | 对 FCoP 的提示 |
|---|---|---|
| Maildir | 单目录 > 10k 文件时 `readdir` 退化 | 百万任务规模前需要分片策略（按日期/thread） |
| procmail | 并发 rename 的 race condition | 多 Agent 并发写同一个 task_id 需要原子性保证 |
| inotify | 单进程 watch 数量硬上限 | 大规模巡检不能全量 watch，需要轮询或摘要 |
| 文件编辑 | 任何人都能手改，破坏 append-only | 需要 Git commit hash 作为外部时间锚 |

Git 的答案是 **packfile + refs + hash tree**——把"文件即协议"转成
"对象即协议"，扛住了规模。FCoP 现在还在 Maildir 阶段，**在撞墙前
就要有对应答案**。当前协议设计刻意规避了这些问题（任务量级低、
append-only 靠约定而非强制），这是早期的**主动选择**，不是遗漏。

## 9. 中立性：FCoP 的护城河

如果 Anthropic / OpenAI / Google 把多 Agent 协作协议内置到各自的
模型能力里，FCoP 这种独立协议还有没有存在的理由？

**有，而且正是因为这个"会被包住"的风险，FCoP 必须把中立性写成
设计原则**：

> **FCoP 永不绑定任何模型、任何平台、任何 IDE。**

三个具体场景使这条原则必须存在：

1. **跨模型协作**：Claude 当 PM + GPT 当 DEV + Gemini 当 QA 的团队
   ——必须有不属于任何厂商的协议
2. **本地 / 离线 / 主权**：企业内网、政府、金融合规环境不可能采用
   厂商内置协议
3. **人类可核对**：厂商内置协作是"两个 AI 私下协调好告诉你结果"；
   FCoP 是"每一步都落成人可审的文件"

这三条场景的共同点是**对中立协议的刚需**。只要这三条之一存在，
FCoP 就有存在空间。

---

## 10. 一句话总结

> **LangGraph 想解决的是"任务怎么稳定跑完"；
> FCoP 想解决的是"这项协作怎么才算真实发生、真实完成、真实可追责"。**
>
> LangGraph 是 Agent 的执行骨架；FCoP 是 Agent 团队的组织语法。
>
> 而组织语法之所以能存在，是因为 FCoP 把协议降到了**文件系统层**——
> **协议先存在，运行时只是消费它**。

## 附录：本篇的反向自校验清单

给未来维护者一个自检清单——如果 FCoP 的某次演进让任何一条答案
变成"否"，就应该停下来重新审视：

- [ ] 没装 `fcop` MCP 的人，用 `ls / cat / mv / git` 还能参与协作吗？
- [ ] 任何新增的 MCP 工具，docstring 里写清了它对应的几条文件操作吗？
- [ ] 协议的核心语义（路由、角色链、回执、归档）还在文件名 + frontmatter +
      目录里吗？还是被藏进了 Python 代码？
- [ ] Solo 模式（1 人 + 1 AI）下，FCoP 还是有独立价值的吗？
- [ ] 换一个 IDE / 换一个模型 / 不用 Cursor，协议还原样工作吗？
- [ ] 协议是否承诺了向后兼容？小版本是否真的没破坏老文件？

六条都是"是"的时候，FCoP 还是 FCoP。任何一条变成"否"——**协议在
往运行时里掉**，应该回头。
