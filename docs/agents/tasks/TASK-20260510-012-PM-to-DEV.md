## §三 现象 + 调查方向（QA 给的线索）

### 3.1 错误信号

```text
[TaskDispatcher] startSession failed for TASK-20260509-999-PM-to-DEV.md:
  agent.send failed for sdk_agent_id="agent-f07388df-...":
  Agent agent-f07388df-... already has active run (code=undefined, isRetryable=false)
```

### 3.2 一致性（高置信度证据）

QA 跑 2 次独立测试（不同 dataDir / 不同 UUID）**100% 复现**：

| 测试 | dataDir | sdk_agent_id | 结果 |
|---|---|---|---|
| 1 | `.smoke-beta1` | `agent-fc838565-8166-4a72-9606-994a9c9b2d47` | already has active run |
| 2 | `.smoke-retry-1223` | `agent-f07388df-ee4c-4b97-ac99-57794658b0f6` | already has active run |

→ 不是 race condition，是确定性 bug。

### 3.3 QA 推测的根因（PM 转引）

> Cursor SDK `Agent.create({ local: { cwd } })` 可能在创建时自动启动一个 run（消化 `prompt` 字段或 fire idle handshake）；`CursorSdkAdapter.send()` 随后调用某种 `agent.start(prompt)` / `agent.send(prompt)` 时 SDK 把它当成「第 2 个 run」拒绝。

### 3.4 调查清单（DEV 自行决定优先级）

1. **看 `@cursor/sdk` 类型定义**：`Agent.create()` 是否有 `prompt` 参数？是否返回 `{ runHandle }` 立即启动 run？是否有 `immediate: false` / `delayedStart` 参数？
2. **DEV-007 P2 EXE spike** 期间应该看过 `node_modules/@cursor/sdk/`，可借力之前的笔记
3. **看 `CursorSdkAdapter.create()` 与 `.send()` 的 SDK 调用细节**：当前是否 `Agent.create({ ... prompt: ... })` 已传 prompt 但 `.send()` 又传 prompt？
4. **看 DEV-002 P1 时的 `agent.send` 实现细节**：`AgentSendSpec.prompt` 当前如何 mapping 到 `@cursor/sdk` 的某个 method（`.send` / `.run` / `.message` / `.continue`）？

## §四 修复方向（PM 不锁死，DEV 用判断）

### 方向 A — 让 `Agent.create()` 不自动启 run

如 SDK 支持 `Agent.create({ ... immediate: false })` 或类似参数，最简洁。

### 方向 B — `CursorSdkAdapter.send()` 复用 `create()` 已启动的 run

如果 `Agent.create({ ..., prompt: spec.prompt })` 已经把 prompt 消化进 run，那 `CursorSdkAdapter.send()` 应：
- **第 1 次** send（紧跟 create 后的第一次）：**不**调 `agent.send`，直接返回 `RunHandle` 复用 create 启动的 run（监听 events 即可）
- **后续** send（同一 sdk_agent_id 多轮对话）：调 `agent.continue(prompt)` 或类似 method

需要在 `CursorSdkAdapter` 内部加状态：每个 `sdk_agent_id` 是否已消费过 create 启动的 run。

### 方向 C — 改 `Agent.create()` **不**传 prompt + `send()` 显式启 run

将 `Agent.create()` 调用裁掉 prompt 字段（如果当前传了的话），让 SDK 不自动启 run；`send()` 就是第 1 个 run。

⚠️ 需确认 SDK `Agent.create()` 是否真有 prompt 参数，否则方向不存在。

### 方向 D — 切 cloud mode（与 BUG-SDK-001 同临时规避）

DEV 实测时如发现 cloud mode 没有 active-run 问题，可作为 **fallback 而不是主修**。仍要给 ADMIN 留 cloud-mode 出口。

**PM 倾向**：方向 A（最简洁）→ 方向 B（最稳健）→ 方向 C（如 SDK 不支持 A）→ 方向 D（fallback only）。

DEV 自己看 SDK 类型再定。**重要**：改完写 1 段「为何选这个方向」到 REPORT §五（同 DEV-010 §五 surprises 风格）。
