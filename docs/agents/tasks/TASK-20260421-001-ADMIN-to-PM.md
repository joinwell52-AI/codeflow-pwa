# 任务：排查并缓解"AI 卡死"的发生路径

## 背景

ADMIN 观察到"AI 卡住"是当前日常使用中频繁出现的情况，希望团队系统性排查、找出真正的卡点，并在合适的位置加上超时/兜底保护。

当前 ADMIN 给不出具体的复现路径——就是"感觉上常常卡"。希望 PM 牵头做一次**分诊（triage）**，把这个模糊感觉拆解到具体的代码路径和可执行的修复项。

## 初始线索（ADMIN 提供的模糊假设）

ADMIN 最初举的例子是"像 `open_panel` 这种可能等待窗口的工具要加 `wait_s` 上限"。但经初步核查：

- `mcp_server.py` 里**不存在** `open_panel` 工具
- 现有所有 MCP 工具都是**本地同步文件操作**，不存在网络/进程等待路径
- 所以"卡死"的真实来源**不在当前 MCP 工具里**

这条线索应被当作 ADMIN 对现象的直觉表达，不是精确的技术诊断。真实的卡点需要 PM 重新排查。

## 任务范围

本次任务**只做分诊，不做大规模修复**。具体包括：

1. 盘点 Bridgeflow 项目里所有**可能阻塞 / 等待 / 挂起**的代码路径，至少覆盖：
   - `codeflow-plugin/scripts/` 下的所有脚本（除 `mcp_server.py` 已查外）
   - `codeflow-desktop/` 桌面端代码里任何窗口/进程/IO 等待点
   - `scripts/` 根目录下运维脚本里的外部命令调用
   - 涉及网络、subprocess、input()、外部进程等待的任何位置
2. 对每个候选位置，标注：
   - 是否真的有卡死风险（还是心理上的"可能会卡"）
   - 有 → 建议的超时值和兜底策略
   - 无 → 明确排除的理由
3. 交付一份**分诊报告**（`REPORT-*` 或 `TASK-*-PM-to-DEV.md` 里附清单），让 DEV 按清单改即可，而不是让 DEV 去猜

## 约束

- **不要**在所有 IO 操作上无差别套 try/except/timeout——这只会让代码看起来"更安全"但实际没解决真卡点
- **只**在真实存在卡死可能的位置加保护
- 如果分诊发现"卡"的感知主要来自 Cursor agent 推理层（而非项目代码），要在报告里明确说明——那不是我们能在这里修的

## 预期交付

- PM 在分诊完成后，要么给出**一份明确的候选清单**（附建议超时值），ADMIN 审批后再派 DEV；要么给出**"项目代码里没有真卡点"的结论**，连同"卡死感知的可能来源"（Cursor agent 层、模型生成层、其他）一起说明。
- 分诊结论作为 `REPORT-20260421-001-PM-to-ADMIN.md` 回执给 ADMIN。
