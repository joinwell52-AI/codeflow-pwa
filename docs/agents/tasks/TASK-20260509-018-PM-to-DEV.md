### 主交付 1：`InboxWatcher`（`packages/codeflow-runtime/src/scheduler/InboxWatcher.ts`，新文件）

按 [`docs/design/codeflow-v2-on-fcop-sdk.md` §2.4](../../docs/design/codeflow-v2-on-fcop-sdk.md) line 1371-1402 reference impl + 本仓现状（task 文件平铺在 `docs/agents/tasks/`，**不**按 role 分子目录）：

```ts
// packages/codeflow-runtime/src/scheduler/InboxWatcher.ts
import chokidar, { type FSWatcher } from "chokidar";

export interface InboxWatcherOpts {
  dir: string;                       // 默认监控 docs/agents/tasks/
  ignoreInitial?: boolean;            // 默认 true（启动时不重复 dispatch 已存在文件）
  fileNamePattern?: RegExp;           // 默认 /^TASK-\d{8}-\d{3}-[A-Za-z]+-to-[A-Za-z]+\.md$/
}

export type InboxEventHandler = (event: InboxEvent) => void | Promise<void>;

export interface InboxEvent {
  kind: "task_added";
  filepath: string;       // 绝对路径
  filename: string;       // 仅文件名
  sender: string;         // 从 filename 解析（如 "ADMIN"）
  recipient: string;      // 从 filename 解析（如 "PM"）
}

export class InboxWatcher {
  constructor(opts: InboxWatcherOpts);

  /** 启动监控；返回 promise，watcher.ready 后 resolve */
  start(): Promise<void>;

  /** 注册事件处理器；返回 unsubscribe */
  onEvent(handler: InboxEventHandler): () => void;

  /** 优雅停止，关闭 fd */
  stop(): Promise<void>;
}
```

**实现要点**：

- 用 `chokidar.watch(dir, { ignoreInitial: true, depth: 0 })`——**只**监控顶级目录（不递归到 `tasks/inbox/<role>/` —— 现状是平铺）
- 事件：只处理 `add` 事件，**不**处理 `change` / `unlink`（task 文件按 codeflow-project 规则只追加不修改）
- 文件名 regex 用任务签名 `/^TASK-\d{8}-\d{3}-[A-Za-z]+-to-[A-Za-z]+\.md$/`——**严格匹配**（`REPORT-*.md` / `HANDOFF-*.md` 都不进 doorbell）
- 解析 sender / recipient：filename 拆分得到 `[..., sender, "to", recipient].md`
- handler 抛错不能拖垮 watcher：用 `Promise.resolve(handler(...)).catch(err => log)` 隔离
- `stop()` 用 chokidar 的 `await watcher.close()` 保证 fd 释放

### 主交付 2：`TaskParser`（`packages/codeflow-runtime/src/scheduler/TaskParser.ts`，新文件）

```ts
import { parse as parseYaml } from "yaml";

export interface ParsedTask {
  filepath: string;
  filename: string;
  frontmatter: Record<string, unknown>;
  body: string;
  // Convenience field accessors:
  task_id?: string;
  sender?: string;
  recipient?: string;
  priority?: "P0" | "P1" | "P2" | "P3";
  thread_key?: string;
  layer?: "worker" | "governance" | "admin";
}

export class TaskParser {
  /** 读取 + 解析，文件不存在 / front-matter 解析失败抛 TaskParseError */
  static async parse(filepath: string): Promise<ParsedTask>;
}
```

**实现要点**：

- front-matter delimiter 用 `^---\r?\n` 严格匹配开头第 1 行 + 第 N 行
- 用 `yaml` 包（npm install 时新增依赖）解析 YAML——**不**自己手写 YAML parser
- 找不到 front-matter（文件不以 `---` 开头）= 仍返回 `frontmatter: {}` + 完整 body（**不抛**——容忍格式不全的文件）
- YAML 解析失败 = throw `TaskParseError`（新错误类，加在 `errors.ts` 主文件，跟 Phase B 决策 J 一致）
- frontmatter 字段映射到顶层 `task_id / sender / ...` 字段时做类型校验——失败则在 `frontmatter` 里保留原值，但顶层字段为 `undefined`（容忍）

### 主交付 3：`StateHistoryWriter`（`packages/codeflow-runtime/src/scheduler/StateHistoryWriter.ts`，新文件）

按 [`§3.3 Task Schema`](../../docs/design/codeflow-v2-on-fcop-sdk.md) line 1611 的 `state_history` 字段语义 + 本仓 task 文件**只追加不修改**的硬约束：

```ts
export interface StateHistoryEntry {
  at: string;              // ISO-8601
  by: string;              // agent_id 或 "runtime" / "ADMIN"
  from: string;            // 状态变更源（如 "inbox" / "running" / "review_pending"）
  to: string;              // 目标状态
  note?: string;           // 可选自由文本（≤ 200 字）
}

export class StateHistoryWriter {
  /** 在指定 task.md 末尾追加一段 state_history 条目（markdown 节）*/
  async append(filepath: string, entry: StateHistoryEntry): Promise<void>;
}
```

**实现要点**：

- 用 `fs.promises.appendFile`（append-only）——**不**改原 front-matter（按 codeflow-project 规则）
- 追加格式（markdown）：

```markdown

---

## state_history (auto-appended by runtime)

- **{ISO at}** | by `{by}` | `{from}` → `{to}` {note 短文本}
```

- 多次 append 时**第二次起**只加 `- bullet`，不重写 `## state_history` 标题——通过读文件检查"是否已含 ## state_history"决定（用 `String.includes` 简单判断即可）
- 文件不存在 = throw `TaskFileNotFoundError`（新错误类）
- 不做 atomic-write（append-only 不需要——丢的就是丢的，下次启动 InboxWatcher 重扫即可）

### 主交付 4：`TaskDispatcher`（`packages/codeflow-runtime/src/scheduler/TaskDispatcher.ts`，新文件）

把 `InboxWatcher + TaskParser + StateHistoryWriter + AgentRegistry + SessionManager` 5 个能力胶合起来：

```ts
export interface TaskDispatcherOpts {
  watcher: InboxWatcher;
  parser?: typeof TaskParser;       // 默认 TaskParser
  historyWriter: StateHistoryWriter;
  registry: AgentRegistry;
  sessionManager: SessionManager;
  logger?: { info(...): void; warn(...): void; error(...): void };
}

export class TaskDispatcher {
  constructor(opts: TaskDispatcherOpts);
  start(): Promise<void>;            // 内部 watcher.start() + watcher.onEvent(...)
  stop(): Promise<void>;
}
```

**Dispatch 流程**（onEvent → 处理）：

```
1. 解析 task.md（用 TaskParser）
   - 解析失败 → log.warn + state_history append "from: inbox / to: parse_failed"
   - 解析成功，进入 step 2

2. 找 recipient agent
   - 从 frontmatter.recipient 找；fallback 到 filename 的 -to-XXX 段
   - 用 AgentRegistry.list({ role: recipient }) 拿对应 agent；找不到 → log.warn + state_history "from: inbox / to: agent_not_found"
   - 找到 = step 3

3. startSession(agent.agent_id, task_id, payload)
   - payload = { task_filepath, task_filename, frontmatter, body }
   - 调 SessionManager.startSession（Phase B 接口已稳定）
   - state_history append "from: inbox / to: dispatched / by: runtime / note: session_id={...}"

4. 监听 session 终结
   - 用 SessionManager.onEvent(handler)，filter 当前 session_id 的 runtime.session_ended / runtime.session_cancelled
   - 终结时 state_history append "from: dispatched / to: {ended|cancelled} / by: runtime / note: cost={...}/duration_ms={...}"
   - 终结后 unsubscribe handler，避免内存泄漏
```

**注意**：Phase C 默认串行——同一 recipient agent 已有 running session 时，**新 task 排队**或**直接 reject**？

- **PM 决定**：默认 **reject + state_history "from: inbox / to: rejected_busy / note: agent already running session={old_id}"**
- 排队（queue）= v0.2 范围（需要更精细的 scheduler）
- ADMIN 后续可手工 cancel 老 session 后再 dispatch 新 task

#### 5.c E2E mini demo：`packages/codeflow-runtime/examples/hello-world.ts`（新目录 + 新文件）

按 [`§0.8.3 v0.1 Hello World Demo`](../../docs/design/codeflow-v2-on-fcop-sdk.md) line 664-707 的脚本：

```ts
// examples/hello-world.ts
import { Runtime } from "@codeflow/runtime";
import { CursorSdkAdapter } from "@codeflow/runtime/adapters";  // 或本 sprint 用 InMemorySdkAdapter

async function main() {
  const runtime = await Runtime.create({
    sdkAdapter: new CursorSdkAdapter(),    // 或 InMemorySdkAdapter（demo 阶段先用 in-memory）
    persistDir: "./.codeflow/state",
    inboxDir: "./examples/inbox",
  });

  await runtime.start();

  // 注册一个 DEV agent
  await runtime.registry.register({
    agent_id: "dev-01",
    layer: "worker",
    runtime: "cursor-sdk@^1.0.12",
    skills: ["fcop", "git"],
  });

  console.log("Runtime started. Drop a TASK-*-XXX-to-dev-01.md in examples/inbox/ to trigger.");
  console.log("Press Ctrl+C to stop.");

  process.on("SIGINT", async () => {
    await runtime.stop();
    process.exit(0);
  });
}

main().catch(console.error);
```

**注意**：本 sprint 用 `InMemorySdkAdapter`（DEV 已在 Phase B 实现）即可——**不需要**真实 Cursor SDK 集成（那是 S6 codeflow-shell 范围）。Demo 的目的是证明"InboxWatcher + Dispatcher + SessionManager + StateHistoryWriter"的链路打通，不是证明 SDK 集成。
