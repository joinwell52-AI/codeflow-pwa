# 巡检器「拍肩膀」技术方案与精准化指南

> 角色：架构 / 产品 / 实现统一说明  
> 目标：在 **无 Cursor 跨会话 API** 的前提下，用桌面自动化 **可靠通知** 各 Agent 窗口「有新任务文件，请处理」。

---

## 一、当前技术路线有没有「根本性问题」

### 1.1 路线是什么

```
任务文件落盘 →（可选）文件系统事件或轮询发现变化
    → 聚焦 Cursor 窗口 → 快捷键切到目标 Agent Tab
    → 打开/确认聊天输入区 → 粘贴固定短句 → Enter
    →（可选）OCR 验证当前角色与输入框
```

这是 **操作系统 + 应用 UI 自动化** 的典型解法，**不是设计错误**：在官方不提供「向指定 Agent 会话投递消息」API 时，没有更「正统」的替代路径。

### 1.2 真正的风险在哪里（不是「该不该用巡检」，而是「边界」）

| 类别 | 说明 |
|------|------|
| **UI 契约不稳定** | Cursor 升级、主题、布局、Pinned/Agents 窗口变化 → 快捷键仍可用，但 OCR 区域、输入框识别可能漂移。 |
| **OCR 本质是非确定性的** | 字号、抗锯齿、中英文混排会导致同一屏识别结果波动 → **不能把 OCR 当唯一真值**，只能当辅助验证或兜底点击。 |
| **焦点与时机** | 切窗、Ctrl+L、粘贴之间存在竞态；机器负载高时 sleep 不足会偶发失败。 |
| **「忙」检测** | 依赖关键词/Stop 按钮等启发式 → 会有假阳性（该催不催）和假阴性（不该催却催）。 |
| **多显示器 / 缩放** | DPI、窗口遮挡会影响截图与坐标。 |

结论：**方案方向正确**；问题集中在 **工程韧性与运维约束**，需要 **分层降级 + 可配置 + 可观测**，而不是推倒重来。

---

## 二、怎样做到「精准、有效」

### 2.1 产品层：先定义「有效」

- **唯一事实源**：协作内容以 `docs/agents/` 下 **TASK 文件** 为准；「拍肩膀」只负责 **唤醒**，不在聊天里重复长正文。  
- **成功标准**：以 **文件被目标角色读取并产生后续动作** 为业务成功；桌面自动化以 **尽最大努力送达短提示** 为技术目标，并允许 **人工补一次点击**。  
- **固定话术**：短句（如「巡检，接收文件」类）降低粘贴失败时的歧义，也减少模型误读长文本。

### 2.2 架构层：快捷键为主，OCR 为辅

1. **第一性路径**：`Ctrl+Alt+1..4`（或 `codeflow.json`（兼容 `CodeFlow.json`）/ `codeflow-nudger.json`（兼容 `codeflow-desktop.json`）配置的键）切 Tab → **成功标准应是「快捷键 + 短延迟」**，而不是「整屏 OCR 认字」。  
2. **OCR 用途收窄**：  
   - 优先用于 **验证当前 Tab 是否为目标角色**（小 ROI 优于全屏）。  
   - 失败时再 **点击角色名兜底**（已有逻辑方向正确）。  
   - 仍不对时由巡检器模拟 **Ctrl+Shift+P**，在命令面板粘贴与 Pinned 一致的标签（如 `2-DEV`）后回车，再 OCR 验证；并与前两步一起做 **多轮重试**，减少焦点丢失或动画未完成导致的假失败。  
3. **避免用 OCR 驱动主路径**：全屏扫字只适合 debug 或兜底，否则延迟和误判都会上去。

### 2.3 实现层：已做 + 建议继续做

**已在代码中体现的方向（示例）**

- 新任务因 **冷却 / Agent 忙 / 发送失败** 未送达时，进入 **待重试队列**，避免「文件已被 watcher 记住却永远不催」（逻辑闭环）。  
- `first_hello` 等模板使用 **安全格式化**，避免无占位符时 `.format` 崩溃。  
- 任务侧 `DEV01` 与界面侧 `DEV` 通过 **角色键归一化** 对齐，减少「已切换仍判失败」。  
- 切换 Agent：`Ctrl+Alt+1..4` → 点击 Tab 名 → **Ctrl+Shift+P + `1-PM`/`2-DEV`…** 三层降级，默认最多 **3 轮**整轮重试，每步后 OCR 再判。

**建议迭代（按性价比排序）**

| 优先级 | 项 | 作用 |
|--------|----|------|
| P0 | **预检面板** 一键检查：项目路径、`docs/agents`、快捷键是否写入 `keybindings.json`、Relay 连通 | 减少「根本没配置好」导致的无效拍肩。 |
| P0 | **固定 Cursor 使用习惯**：单窗口、Agents 栏可见、避免多 Cursor 实例抢焦点 | 运维约束比改代码更能提成功率。 |
| P1 | **文件监听**：用 `watchdog`（或 Windows `ReadDirectoryChangesW`）替代纯轮询，缩短「落盘→第一次尝试拍肩」延迟，并降低空转 | 实现时注意编辑器保存的多次写入，需 **去抖**。 |
| P1 | **可配置延迟**：`codeflow-nudger.json`（兼容旧名 `codeflow-desktop.json`）中 `post_hotkey_delay_ms`、`after_ctrl_l_delay_ms` 等，适配慢机器 | 精准 often = 可调。 |
| P2 | **ROI OCR**：只对 Tab 条或输入框附近截图识别，减少干扰 | 提升准确率、降延迟。 |
| P2 | **结构化日志 + 面板轨迹**：`patrol_trace` 阶段键 + `/api/patrol_trace` + 面板「巡检轨迹」表，与 `[巡检]` 日志同源 | 精确知道巡检器做了什么。 |

### 2.4 指标：怎么知道「变准了」

建议在桌面日志或面板中长期看：

- **拍肩成功率** = 发送成功次数 / 尝试次数（区分「窗口未找到」「切换失败」「粘贴失败」）。  
- **重试命中率** = 第 1 次失败、第 N 次成功占比。  
- **从文件落盘到首次尝试** 的延迟分布（引入 watchdog 后应下降）。

无指标则只能凭感觉，难以迭代。

## 三、与 MCP、主 Agent 窗口的关系（边界清晰）

- **MCP**：便于在 **当前会话** 内用工具读写任务文件；**不能**替代「切到另一个 Agent 窗口拍肩」。  
- **Cursor 主 Agent / Subagents**：适合 **单用户、单产品内的任务拆分**；**码流（CodeFlow）** 的 **多角色人格、文件留痕、手机 ADMIN、中继** 仍是独立价值。  
- **拍肩膀** 与上述两者 **互补**：文件协议是「说什么」，巡检是「谁去听」——在 Cursor 未开放跨会话 API 前，这一层仍需要。

---

## 四、结论摘要

1. **技术方案**：在现有约束下 **合理**，问题主要在 **UI 漂移、OCR 非确定性、时机与焦点**，需用 **分层策略 + 重试 + 配置化 + 预检** 扛住，而不是否定「拍肩膀」本身。  
2. **精准有效**：产品上以 **文件为唯一事实源 + 短句唤醒**；工程上以 **快捷键为主、OCR 验证与兜底为辅、待重试队列防丢、可观测与可调延迟** 持续收紧误差带。
